Long-Term Strategy Rules & Statistical Rationale

Formulated directly from mined production logs (42 trades analyzed) to eliminate transaction fee drag, filter low-expectancy environments, and establish steady walk-forward growth.

Rule 1: Trend Filter Alignment (Kalman Filter) Only execute Buy/Long trades when the Kalman filter indicates a solid up-trend. Rationale: Production logs reveal that Kalman trend-aligned entries (Value 1.0) achieved an expectancy of $-0.0064, while trend-opposing entries (Value -1.0) crashed to $-0.0628 (nearly 10x higher average loss!).
Rule 2: Extended Holding Horizon (4-Hour Minimum) Set minimum trade holding duration to 240 minutes (4 hours) or hold until ATR targets are crossed, up to 48 hours. Rationale: Mined logs show an average holding time of 87.81 minutes. At this frequency, standard exchange transaction fees (0.15%) and bid-ask slippage consume the entirety of the returns. Moving to a multi-hour horizon lets the trend outrun execution costs.
Rule 3: Volatility-Targeted Position Sizing Dynamically scale positions by Target Volatility (1.5% of price) / Current ATR. Rationale: Volatility clustering causes outsized stops during peak volatility, leading to massive drawdowns. Sizing inversely to Average True Range standardizes risk contribution across all regimes.
Rule 4: Neural Network Consensus Gate Restrict execution of long positions if the active Policy brain outputs a weight below 12% for the Kalman filter trend. Rationale: Mined results show that when the NN allocation consensus drops (Value -1.0), average trade loss was $-0.0643. When positive, average loss was constrained to $-0.0178 (a **72.3% reduction**).
Rule 5: Multi-Layer Drawdown Circuit Breaker Halt new position triggers if the main trading system drawdown exceeds 5% or the shadow walk-forward model drawdown exceeds 10%. Rationale: Protects capital against structural market regime changes or API data feed corruptions.

Architecture — What Actually Runs

What follows is the real architecture. No imaginary daemons or fake cron agents. These are the actual source files, their runtime responsibilities, and their data flows.

PER-TICK RUNTIME FLOW (process_tick, called ~1-10s per ticker in live mode)
data_ingestion.py → Fetch price from CCXT / CSV. Compute RSI, MACD, BB, ATR.
└─ row = {"close", "rsi", "macd", "bb_upper", "bb_lower", "atr", ...}
main.py (process_tick) →
├─ quant_utils.estimate_ou_process() → OU theta + regime (trending/mr)
├─ learning_engine.get_state_vector() → 8D vector [rsi, macd, bb, atr, ou_theta, ...]
├─ learning_engine.select_weights(state) → PolicyNetwork softmax → 12 weights
├─ strategy_engine.get_weighted_signal() → weighted signal (-1..1)
├─ 12 strategies : ema_cross, rsi_reversal, bb_reversion, kalman, stophunt, ml_rf, momentum, volatility, mean_reversion, trend, scalping, breakout
├─ probability_engine.evaluate_trade() → p_win, EV, Kelly fraction, is_viable
├─ evaluation.safety.KillSwitch.check() → blocks if drawdown/position/exposure limits exceeded
├─ execution_engine.open_position() → CCXT order (live) or simulated fill (paper/sim)
├─ execution_engine.update_positions() → check SL/TP every tick
└─ broadcast_message() → WebSocket push to dashboard
ON TRADE CLOSE (on_trade_closed callback)
├─ evaluation.safety.KillSwitch.record_trade(pnl)
├─ evaluation.safety.DrawdownTracker.update(equity)
├─ strategy_engine.record_trade_outcome() → update rolling win rates
└─ learning_engine.learn_from_trade() → PolicyNetwork backprop
data_ingestion.py

Fetches historical/candles from CCXT (Kraken). Computes RSI, MACD, Bollinger Bands, ATR. Runs live WebSocket streams via ccxt.pro. Optional CSV reload for sim.

strategy_engine.py

12 strategy classes with per-strategy signals. Ensemble weights from PolicyNetwork. OU regime detection. Performance biasing (rolling win rate).

learning_engine.py

Policy network (4-layer MLP: 8→12→12→12). REINFORCE-like backprop with PnL reward. State vector builder. Brain persistence in DB. Weight migration 7→12.

execution_engine.py

Position management (open/close/update). Live order routing via CCXT. SL/TP tracking. Cooldown system. Balance persistence to DB.

probability_engine.py

ATR-based SL/TP. Win prob from signal + RSI + historical similarity. Kelly sizing. Drawdown-aware + calibration-capped position sizing.

quant_utils.py

OU process estimation (OLS on log prices), regime detection, signal normalization utilities.

evaluation/ package

safety.py (KillSwitch, DrawdownTracker, MutationFreeze), metrics.py (Sharpe, Sortino, Kelly), cost_model.py, harness.py, position_sizing.py, singletons.py

database.py

SQLite WAL. Tables: settings, trades, portfolio_history, policy_brains, active_assets, notifications, backups, agent_runs, optimizations.

main.py

FastAPI server. Orchestrator class. Tick processing loop. WebSocket broadcast. REST API + dashboard serving. Startup/shutdown hooks.

Current Limitations

  • Policy network is an MLP, not a deep RL agent — weights shift slowly, no replay buffer
  • No GPU support — everything is CPU-only (scikit-learn, numpy)
  • Live exchange WebSocket feeds disconnect periodically (ccxt.pro reconnection)
  • Kraken API balance is ~$89 equity / $4.81 cash — most trades hit "Insufficient funds"
  • No historical backtest framework integrated with live pipeline (backtest_engine.py is standalone)
  • MutationFreeze blocks agent config changes — acts as a safety gate, not a learning loop
  • Sparse documentation — dashboard has inline docs, code has docstrings, no external wiki
